# Домашнее задание к занятию «Микросервисы: принципы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.

## Задача 1: API Gateway

Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.

Решение должно соответствовать следующим требованиям:
- маршрутизация запросов к нужному сервису на основе конфигурации,
- возможность проверки аутентификационной информации в запросах,
- обеспечение терминации HTTPS.

Обоснуйте свой выбор.

**Ответ:**  
**Мой выбор Kong Gateway**  
Сравнительная таблица решений  

| Решение | Маршрутизация | Проверка аутентификации | Терминация HTTPS | Особенности |
|---------|---------------|-------------------------|------------------|-------------|
| **NGINX (OpenResty)** | Да (на основе `location`, `proxy_pass`) | Да (модуль `auth_request`, JWT) | Да (директива `ssl`) | Высокая производительность, но для сложной логики требует написания Lua-скриптов |
| **Kong Gateway** | Да (гибкая настройка Routes/Services) | Да (плагины: JWT, OAuth2, OIDC, LDAP) | Да (централизованное управление сертификатами) | Построен на NGINX. Богатая экосистема плагинов, динамическая конфигурация через Admin API |
| **Traefik** | Да (Routers, Rules) | Да (ForwardAuth, встроенные плагины) | Да (автоматический выпуск Let's Encrypt) | Отличная интеграция с Docker/K8s, автообнаружение сервисов |
| **Tyk** | Да (API Definitions) | Да (встроенные плагины аутентификации) | Да (TLS termination) | Написан на Go, имеет встроенный дашборд, поддерживает GraphQL |

Обоснование выбора: Kong Gateway  
Kong Gateway является зрелым и стандартизированным решением для API Gateway в микросервисной архитектуре  
Он построен на NGINX, что гарантирует высокую производительность и надежную терминацию HTTPS  
Главное преимущество Kong — мощная экосистема плагинов, которая позволяет "из коробки" реализовать проверку аутентификации (JWT, OAuth2), ограничение скорости (Rate Limiting), логирование и трансформацию запросов  
Маршрутизация гибко настраивается на основе путей, заголовков и HTTP-методов  
Наличие Admin API позволяет динамически управлять конфигурацией без перезагрузки сервиса, что критично для микросервисов  


## Задача 2: Брокер сообщений

Составьте таблицу возможностей различных брокеров сообщений. На основе таблицы сделайте обоснованный выбор решения.

Решение должно соответствовать следующим требованиям:
- поддержка кластеризации для обеспечения надёжности,
- хранение сообщений на диске в процессе доставки,
- высокая скорость работы,
- поддержка различных форматов сообщений,
- разделение прав доступа к различным потокам сообщений,
- простота эксплуатации.

Обоснуйте свой выбор.

**Ответ:**  
**Мой выбор RabbitMQ**  
Сравнительная таблица решений  

| Решение | Кластеризация | Хранение на диске | Скорость | Форматы сообщений | Права доступа (ACL) | Простота эксплуатации |
|---------|---------------|-------------------|----------|-------------------|---------------------|----------------------|
| **RabbitMQ** | Да (Quorum queues, Mirrored queues) | Да | Высокая | AMQP, MQTT, STOMP, HTTP | Да (vhosts, granular permissions) | Очень высокая |
| **Apache Kafka** | Да (Consumer groups, репликация) | Да | Очень высокая | Любые (байты/сериализация) | Да (ACL) | Средняя |
| **Apache Pulsar** | Да (Apache BookKeeper) | Да | Очень высокая | Любые | Да (тенанты, ACL) | Низкая |
| **NATS JetStream** | Да (Raft clustering) | Да | Очень высокая | Любые | Да (Accounts) | Высокая |

Обоснование выбора: RabbitMQ  
RabbitMQ идеально балансирует все заявленные требования.
Он поддерживает надежную кластеризацию (Quorum queues для высокой доступности), сохраняет сообщения на диск и обеспечивает высокую скорость доставки  
Ключевое преимущество — поддержка множества протоколов и форматов сообщений "из коробки" (AMQP, MQTT, STOMP) благодаря плагинам  
RabbitMQ обладает гибкой и понятной системой разделения прав доступа (виртуальные хосты, права пользователей на конкретные очереди/топики)  
По сравнению с Kafka или Pulsar, которые ориентированы на стриминг огромных объемов данных и сложнее в настройке  
RabbitMQ значительно проще в эксплуатации, имеет понятный UI (Management Plugin) и покрывает 99% задач межсервисного взаимодействия
