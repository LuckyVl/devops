# Домашнее задание к занятию «Микросервисы: подходы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

Обоснуйте свой выбор.

**Ответ:**  
**Мой выбор решения** - GitLab в связке с HashiCorp Vault для хранения секретов и GitLab Runner для агентов сборки  

**Альтернатива** - GitHub + GitHub Actions, но GitLab исторически предлагает более гибкую работу с раннерами и шаблонами из коробки  

**Обоснование и архитектура**  

**Хранение кода и Git**
- GitLab предоставляет полноценный Git-хостинг  
- Для микросервисов используем подход каждый микросервис в отдельном репозитории, что позволяет командам работать автономно  

**CI/CD (GitLab CI/CD)** 
- **запуск по событию** - настраивается через директиву rules или only/changes
- **запуск по кнопке с параметрами** - используется директива when: manual
- **настройки к каждой сборке и шаблоны** - GitLab поддерживает локальные, удаленные файлы, шаблоны проектов, это позволяет вынести общие шаги в централизованный репозиторий шаблонов
- **несколько конфигураций из одного репозитория:** - можно собирать один сервис с разными версиями зависимостей или для разных окружений
- **кастомные шаги и Docker-образы** - любой шаг может выполняться в кастомном образе, указанном в директиве image

**Агенты сборки (GitLab Runner):**

- **собственные серверы** - GitLab Runner устанавливается на серверы компании, можно использовать Docker или Kubernetes executor для изоляции сборок
- **параллельные запуски** - настраивается в конфигурации раннера
- **параллельные тесты** - в самом пайплайне тесты разбиваются на джобы, которые выполняются параллельно


**Безопасное хранение секретов:**

- **HashiCorp Vault** - GitLab имеет нативную интеграцию с Vault


## Задача 2: Логи

Предложите решение для обеспечения сбора и анализа логов сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор логов в центральное хранилище со всех хостов, обслуживающих систему;
- минимальные требования к приложениям, сбор логов из stdout;
- гарантированная доставка логов до центрального хранилища;
- обеспечение поиска и фильтрации по записям логов;
- обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов;
- возможность дать ссылку на сохранённый поиск по записям логов.

Обоснуйте свой выбор.

**Ответ:**  
**Мой выбор Vector + Elasticsearch + Kibana**

**Обоснование и архитектура**

- **Минимальные требования к приложениям** - микросервисы просто пишут логи в stdout
- **Сбор и Гарантированная доставка** - Vector развертывается на каждом хосте или как DaemonSet. Он читает stdout контейнеров. Vector поддерживает буферизацию на диск, что гарантирует доставку логов даже при падении сети или перезапуске Elasticsearch. 
- **Центральное хранилище** - Elasticsearch индексирует логи, обеспечивая быстрый полнотекстовый поиск 
- **UI** - Kibana предоставляет интерфейс для поиска и фильтрации. С помощью Kibana Spaces и RBAC можно дать разработчикам доступ только к логам их сервисов 
- **Поисковые ссылки** - Kibana умеет генерировать постоянные ссылки на текущий запрос с учетом фильтров и временного диапазона


## Задача 3: Мониторинг

Предложите решение для обеспечения сбора и анализа состояния хостов и сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор метрик со всех хостов, обслуживающих систему;
- сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network;
- сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network;
- сбор метрик, специфичных для каждого сервиса;
- пользовательский интерфейс с возможностью делать запросы и агрегировать информацию;
- пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы.

Обоснуйте свой выбор.

**Ответ:**  
**Мой выбор Prometheus + Grafana + Node Exporter + cAdvisor**  

**Обоснование и архитектура**

- **Метрики хостов (CPU, RAM, HDD, Network)** - на каждый хост устанавливается Node Exporter, который отдает метрики по протоколу HTTP 
- **Метрики ресурсов сервисов (контейнеров)** - cAdvisor анализирует использование ресурсов (CPU, RAM, Network I/O) каждым запущенным Docker-контейнером 
- **Специфичные метрики сервисов** - микросервисы используют клиентские библиотеки для экспорта бизнес-метрик 
- **Сбор метрик** - Prometheus работает по pull-модели, это практичнее для микросервисов, где IP-адреса могут динамически меняться
- **UI и Дашборды** - Grafana подключается к Prometheus как источнику данных, возможна гибкая настройка данных и графиков
